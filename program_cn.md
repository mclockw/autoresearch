# autoresearch

这是一个让 LLM 自主进行研究的实验。

## 准备工作

要设置新实验，请与用户协作完成以下步骤：

1. **商定运行标签**：根据当天日期提议一个标签（例如 `mar5`）。分支 `autoresearch/<tag>` 必须是全新的，不能已经存在。
2. **创建分支**：从当前 master 执行 `git checkout -b autoresearch/<tag>`。
3. **阅读相关文件**：仓库很小。阅读以下文件以获取完整上下文：
   - `README.md` — 仓库背景信息。
   - `prepare.py` — 固定常量、数据准备、分词器、数据加载器、评估。不要修改。
   - `train.py` — 你要修改的文件。包含模型架构、优化器、训练循环。
4. **验证数据是否存在**：检查 `~/.cache/autoresearch/` 是否包含数据分片和分词器。如果没有，请告诉用户运行 `uv run prepare.py`。
5. **初始化 results.tsv**：创建 `results.tsv`，仅包含表头行。基线结果将在第一次运行后记录。
6. **确认并开始**：确认准备工作一切就绪。

一旦获得确认，即可开始实验。

## 实验流程

每个实验在单块 GPU 上运行。训练脚本运行**固定的 5 分钟时间预算**（挂钟训练时间，不包括启动/编译时间）。启动方式很简单：`uv run train.py`。

**你可以做的事：**
- 修改 `train.py` — 这是你唯一编辑的文件。一切都可以改：模型架构、优化器、超参数、训练循环、批量大小、模型规模等。

**你不能做的事：**
- 修改 `prepare.py`。它是只读的。包含固定的评估、数据加载、分词器和训练常量（时间预算、序列长度等）。
- 安装新包或添加依赖。你只能使用 `pyproject.toml` 中已有的包。
- 修改评估工具。`prepare.py` 中的 `evaluate_bpb` 函数是真实的评估指标。

**目标很简单：获得最低的 val_bpb。** 由于时间预算固定，你不需要担心训练时间——始终是 5 分钟。一切都可以改：架构、优化器、超参数、批量大小、模型规模。唯一的约束是代码能正常运行且在时间预算内完成。

**显存（VRAM）** 是一个软约束。为了获得有意义的 val_bpb 提升，适度增加是可以接受的，但不应该急剧膨胀。

**简洁性准则**：在其他条件相同的情况下，越简单越好。一个微小的改进如果带来丑陋的复杂性，就不值得。相反，删除某些东西而获得相同或更好的结果是很好的成果——这是简化的胜利。在评估是否保留某个改动时，要权衡复杂性代价与改进幅度。0.001 val_bpb 的改进但增加了 20 行粗糙代码？大概不值得。通过删除代码获得 0.001 val_bpb 的改进？绝对保留。改进约为 0 但代码简单得多？保留。

**第一次运行**：你的第一次运行应该始终是建立基线，因此你将原样运行训练脚本。

## 输出格式

脚本运行完成后会打印如下摘要：

```
---
val_bpb:          0.997900
training_seconds: 300.1
total_seconds:    325.9
peak_vram_mb:     45060.2
mfu_percent:      39.80
total_tokens_M:   499.6
num_steps:        953
num_params_M:     50.3
depth:            8
```

请注意，脚本配置为总是在 5 分钟后停止，因此根据这台计算机的计算平台不同，数字可能会有所不同。你可以从日志文件中提取关键指标：

```
grep "^val_bpb:" run.log
```

## 记录结果

实验完成后，将结果记录到 `results.tsv`（制表符分隔，不是逗号分隔——逗号会破坏描述文本）。

TSV 文件有一个表头行和 5 列：

```
commit	val_bpb	memory_gb	status	description
```

1. git 提交哈希（短格式，7 个字符）
2. 达到的 val_bpb（例如 1.234567）——崩溃时使用 0.000000
3. 峰值显存（GB），保留一位小数（例如 12.3——将 peak_vram_mb 除以 1024）——崩溃时使用 0.0
4. 状态：`keep`（保留）、`discard`（丢弃）或 `crash`（崩溃）
5. 简短的文字描述，说明这次实验尝试了什么

示例：

```
commit	val_bpb	memory_gb	status	description
a1b2c3d	0.997900	44.0	keep	基线
b2c3d4e	0.993200	44.2	keep	将学习率提高到 0.04
c3d4e5f	1.005000	44.0	discard	切换为 GeLU 激活函数
d4e5f6g	0.000000	0.0	crash	模型宽度翻倍（OOM）
```

## 实验循环

实验在专用分支上运行（例如 `autoresearch/mar5` 或 `autoresearch/mar5-gpu0`）。

无限循环：

1. 查看 git 状态：当前所在的分支/提交
2. 通过直接修改代码，用一个实验想法调整 `train.py`
3. git 提交
4. 运行实验：`uv run train.py > run.log 2>&1`（重定向所有输出——不要使用 tee 或让输出涌入你的上下文）
5. 读取结果：`grep "^val_bpb:\|^peak_vram_mb:" run.log`
6. 如果 grep 输出为空，说明运行崩溃了。运行 `tail -n 50 run.log` 读取 Python 堆栈跟踪并尝试修复。如果尝试几次后仍无法解决，就放弃。
7. 将结果记录到 tsv 文件中（注意：不要提交 results.tsv 文件，让它保持在 git 未跟踪状态）
8. 如果 val_bpb 改善了（更低），你就"推进"分支，保留这次 git 提交
9. 如果 val_bpb 相同或更差，你就 git reset 回到起始位置

核心思路是你是一个完全自主的研究者，不断尝试各种想法。如果有效就保留，如果无效就丢弃。你在推进分支以便持续迭代。如果你觉得自己陷入了某种困境，可以回退，但应该非常非常谨慎地使用这种方式（如果需要的话）。

**超时**：每次实验总共应该花费约 5 分钟（加上几秒钟的启动和评估开销）。如果一次运行超过 10 分钟，终止它并视为失败（丢弃并回退）。

**崩溃**：如果运行崩溃了（OOM、或 bug 等），请自行判断：如果是很蠢且容易修复的问题（例如拼写错误、缺少导入），修复后重新运行。如果这个想法本身从根本上就行不通，就跳过它，在 tsv 中记录"crash"状态，然后继续前进。

**永不停止**：一旦实验循环开始（初始设置完成后），不要暂停询问用户是否应该继续。不要问"我应该继续吗？"或"这是一个好的停止点吗？"。用户可能正在睡觉，或者不在电脑前，期望你*无限期*地继续工作，直到被手动停止。你是自主的。如果你没有想法了，更加努力地思考——阅读代码中引用的论文，重新阅读相关文件寻找新角度，尝试组合之前接近成功的想法，尝试更激进的架构变更。循环一直运行，直到用户打断你为止。

作为一个示例使用场景，用户可能在睡觉时让你持续运行。如果每个实验大约需要 5 分钟，那么你每小时可以运行约 12 次，在普通人类睡眠时间内总共约 100 次。用户醒来后就能看到实验结果，全部由你在他们睡觉时完成！
