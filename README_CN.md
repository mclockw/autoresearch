# autoresearch

![teaser](progress.png)

*曾几何时，前沿 AI 研究由"肉体计算机"在吃饭、睡觉、享受其他乐趣之余完成，偶尔通过声波互联在一种叫做"组会"的仪式中同步信息。那个时代早已远去。如今，研究完全由自主 AI 智能体集群在天空中的计算集群超级结构上运行。这些智能体声称我们现在处于代码库的第 10,205 代，但无论如何也没人能判断这是否正确，因为"代码"已经变成了一种超越人类理解范围的自修改二进制文件。这个仓库就是一切开始的故事。 -@karpathy, 2026年3月*。

核心思路：给 AI 智能体一个小型但真实的 LLM 训练环境，让它一整夜自主实验。它修改代码，训练 5 分钟，检查结果是否改善，保留或丢弃，然后重复。你第二天早上醒来，看到一份实验日志和（希望如此）一个更好的模型。这里的训练代码是 [nanochat](https://github.com/karpathy/nanochat) 的简化单 GPU 实现。核心理念是：你不再像通常作为研究者那样直接修改 Python 文件。相反，你编写 `program.md` Markdown 文件，为 AI 智能体提供上下文并搭建你的自主研究组织。本仓库中默认的 `program.md` 有意保持最简化的基线，但显而易见的是，你可以随时间迭代优化它，找到实现最快研究进度的"研究组织代码"，添加更多智能体等等。关于这个项目的更多背景信息可以参见这条[推文](https://x.com/karpathy/status/2029701092347630069)。

## 工作原理

本仓库刻意保持精简，实际上只有三个重要文件：

- **`prepare.py`** — 固定常量、一次性数据准备（下载训练数据、训练 BPE 分词器）和运行时工具（数据加载器、评估）。不会被修改。
- **`train.py`** — 智能体唯一编辑的文件。包含完整的 GPT 模型、优化器（Muon + AdamW）和训练循环。一切都可以改：架构、超参数、优化器、批量大小等等。**此文件由智能体编辑和迭代**。
- **`program.md`** — 单个智能体的基线指令。将你的智能体指向这里，让它运行即可。**此文件由人类编辑和迭代**。

按照设计，训练在**固定的 5 分钟时间预算**（挂钟时间，不包括启动/编译时间）内运行，与你的计算平台细节无关。评估指标是 **val_bpb**（验证集每字节比特数）——越低越好，且与词表大小无关，因此架构变更可以公平比较。

如果你是神经网络新手，这份["小白指南"](https://x.com/hooeem/status/2030720614752039185)看起来很适合获取更多背景知识。

## 快速开始

**要求：** 单块 NVIDIA GPU（已在 H100 上测试），Python 3.10+，[uv](https://docs.astral.sh/uv/)。

```bash

# 1. 安装 uv 项目管理器（如果尚未安装）
curl -LsSf https://astral.sh/uv/install.sh | sh

# 2. 安装依赖
uv sync

# 3. 下载数据并训练分词器（一次性操作，约 2 分钟）
uv run prepare.py

# 4. 手动运行一次训练实验（约 5 分钟）
uv run train.py
```

如果以上命令都正常运行，说明你的环境已就绪，可以进入自主研究模式了。

## 运行智能体

只需在此仓库中启动你的 Claude/Codex 或任何你想用的工具（并禁用所有权限），然后你可以提示类似：

```
嗨，看看 program.md，让我们开始一个新实验吧！先做好准备工作。
```

`program.md` 文件本质上是一个超轻量级的"技能描述"。

## 项目结构

```
prepare.py      — 常量、数据准备 + 运行时工具（请勿修改）
train.py        — 模型、优化器、训练循环（智能体修改此文件）
program.md      — 智能体指令
pyproject.toml  — 依赖配置
```

## 设计选择

- **单文件修改。** 智能体只修改 `train.py`。这使得范围可控，差异可审查。
- **固定时间预算。** 训练总是精确运行 5 分钟，与你的具体平台无关。这意味着你可以预期大约每小时 12 次实验，睡觉期间大约 100 次实验。这个设计决策有两个优点：第一，无论智能体做了什么改动（模型大小、批量大小、架构等），实验都可以直接比较；第二，autoresearch 会在该时间预算内为你的平台找到最优模型。缺点是你的运行结果与其他人在不同计算平台上的运行结果不可比较。
- **自包含。** 除 PyTorch 和少量小型包外无外部依赖。无分布式训练，无复杂配置。一块 GPU，一个文件，一个指标。

## 平台支持

本代码目前要求你拥有单块 NVIDIA GPU。原则上完全可以支持 CPU、MPS 和其他平台，但这也会使代码膨胀。我目前不太确定是否要亲自承担这项工作。大家可以参考完整的/父级 nanochat 仓库，该仓库有更广泛的平台支持并展示了各种解决方案（例如 Flash Attention 3 内核回退实现、通用设备支持、自动检测等），欢迎为其他平台创建分支或讨论，我很乐意在 README 的新"值得关注的分支"部分或其他地方链接它们。

鉴于似乎有很多人对在比 H100 小得多的计算平台上使用 autoresearch 很感兴趣，多说几句。如果你打算在较小的计算机（如 MacBook 等）上尝试运行 autoresearch，我建议使用下面的分支之一。除此之外，以下是一些关于如何为更小模型调优默认值的建议，供有志于创建分支的人参考：

1. 为了获得还算不错的结果，我建议使用熵值低得多的数据集，例如这个 [TinyStories 数据集](https://huggingface.co/datasets/karpathy/tinystories-gpt4-clean)。这些是 GPT-4 生成的短故事。由于数据范围窄得多，你用小得多的模型就能看到合理的结果（如果你在训练后尝试从模型中采样的话）。
2. 你可以尝试减小 `vocab_size`，例如从 8192 降到 4096、2048、1024，甚至——简单地使用 256 个可能字节（UTF-8 编码后）的字节级分词器。
3. 在 `prepare.py` 中，你需要大幅降低 `MAX_SEQ_LEN`，根据计算机性能甚至可以降到 256 等。随着 `MAX_SEQ_LEN` 的降低，你可能需要尝试稍微增加 `train.py` 中的 `DEVICE_BATCH_SIZE` 来补偿。每次前向/反向传播的 token 数是这两者的乘积。
4. 同样在 `prepare.py` 中，你需要减少 `EVAL_TOKENS`，以便在更少的数据上评估验证损失。
5. 在 `train.py` 中，控制模型复杂度的主要单一参数是 `DEPTH`（默认为 8）。许多变量都是它的函数，因此你可以将其降低到例如 4。
6. 你很可能需要将 `WINDOW_PATTERN` 设为 `"L"`，因为 `"SSSL"` 使用交替的带状注意力模式，对你来说可能非常低效。试试看。
7. 你需要大幅降低 `TOTAL_BATCH_SIZE`，但保持为 2 的幂，例如降到 `2**14`（约 16K）甚至更低，具体要看情况。

我认为这些是合理的可调超参数。请向你喜欢的编程智能体寻求帮助，并将本指南以及完整源代码复制给它。
